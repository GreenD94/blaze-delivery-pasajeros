📋 Estándares de Documentación

Guía para documentar problemas y soluciones en Blaze Delivery Pasajeros

📁 Estructura de la Documentación

La documentación de problemas está organizada de la siguiente manera:

📋 lista-problemas.html

Archivo principal que muestra una vista general de todos los problemas identificados, con estadísticas y enlaces a cada issue individual.

📋 estandares-documentacion.html

Este archivo que contiene las reglas y estándares para documentar problemas correctamente.

📁 issues/

Carpeta que contiene cada problema documentado en archivos HTML individuales.

🎨 styles.css

Archivo de estilos compartido por todos los documentos para mantener consistencia visual.

📂 Organización de Archivos

documentation/issues-to-resolve/
├── lista-problemas.html          # Vista general de problemas
├── estandares-documentacion.html # Este archivo
├── styles.css                    # Estilos compartidos
└── issues/                       # Carpeta de problemas individuales
    ├── scaffold-context-error.html
    ├── [nuevo-problema].html
    └── [mas-problemas].html

🧭 Reglas de Navegación

Cada archivo de la documentación debe incluir enlaces de navegación apropiados:

📋 lista-problemas.html

Debe incluir enlaces a:

  • ✅ estandares-documentacion.html
  • ✅ Cada issue individual en issues/

📋 estandares-documentacion.html

Debe incluir enlaces a:

  • ✅ lista-problemas.html

📁 issues/[problema].html

Cada issue individual debe incluir enlaces a:

  • ✅ lista-problemas.html
  • ✅ estandares-documentacion.html

🔗 Rutas Relativas

Usar las rutas relativas correctas según la ubicación del archivo:

# Desde issues/[problema].html:
../lista-problemas.html
../estandares-documentacion.html
../styles.css

# Desde archivos en la raíz:
lista-problemas.html
estandares-documentacion.html
styles.css
issues/[problema].html

📊 Estados de los Problemas

Cada problema documentado debe tener uno de los siguientes estados:

🔄 Pendiente

Problema identificado pero aún no se ha iniciado el trabajo

📝 Por Hacer

Problema planificado y listo para ser trabajado

⚡ En Curso

Problema actualmente siendo trabajado por un desarrollador

👀 En Revisión

Solución implementada y pendiente de revisión

✅ Listo

Problema resuelto y verificado

¿Cómo documentar un problema?

1. Ubicación del Problema

Especificar exactamente dónde se encuentra el problema: archivo, línea, función, etc.

2. Descripción Detallada

Explicar claramente de qué se trata el problema y qué comportamiento se espera vs. el actual.

3. Solución Recomendada

Proporcionar una solución específica con código de ejemplo cuando sea posible.

4. Ventajas de la Solución

Listar los beneficios y mejoras que aporta la solución propuesta.

5. Código de Ejemplo

Incluir bloques de código que muestren el problema y la solución.

6. Diagramas/Visualizaciones

Usar diagramas cuando sea necesario para explicar conceptos complejos.

📝 Estructura de un Issue Individual

Cada archivo de issue en la carpeta issues/ debe seguir esta estructura:

1. Header con título del problema
2. Enlaces de navegación (lista-problemas.html y estandares-documentacion.html)
3. Ubicación del problema (archivo, línea, función)
4. Descripción detallada del problema
5. Solución recomendada con código
6. Ventajas de la solución
7. Diagramas o visualizaciones (si aplica)
8. Fecha de identificación y reportero
9. Botón de formato Jira
10. Modal de formato Jira

📋 Campos Obligatorios

  • Título del problema: Descriptivo y específico
  • Estado: Uno de los 5 estados definidos
  • Ubicación: Archivo, línea y función específica
  • Descripción: Explicación clara del problema
  • Solución: Código y explicación de la corrección
  • Enlaces de navegación: A lista-problemas.html y estandares-documentacion.html

🪟 Modal de Formato Jira

Para cada problema documentado, se debe incluir un botón que abra un modal de formato Jira con la siguiente funcionalidad:

  • Mostrar el título y la descripción del problema de forma separada.
  • Botón para copiar solo el título.
  • Botón para copiar solo la descripción.
  • Visualización de los campos sugeridos para Jira (sin botón de copiado).
  • El modal debe poder cerrarse con el botón de cierre, la tecla Escape o clic fuera (light dismiss).
  • El modal debe estar implementado usando el elemento <dialog> nativo de HTML.

🎨 Estilos y Consistencia Visual

Todos los archivos deben usar el archivo styles.css compartido para mantener:

  • Consistencia visual: Mismos colores, fuentes y espaciado
  • Responsive design: Funcionamiento en móviles y desktop
  • Reutilización de código: Evitar duplicación de estilos
  • Mantenibilidad: Cambios centralizados en un solo archivo

🔀 Estándares para ramas, commits y merge en Git

📌 Nombres de ramas

  • El nombre de la rama debe seguir el formato: nombre-del-programador/nombre-de-la-caracteristica-o-problema
  • Ejemplo: juan/scaffold-context-error-fix
# Crear y cambiar a la nueva rama
git checkout -b juan/scaffold-context-error-fix

✍️ Commits

  • Los mensajes de commit deben ser resumidos y concisos, describiendo claramente el cambio realizado.
  • Ejemplo: Corrige error de contexto de Scaffold usando GlobalKey en RiderDashBoardScreen
# Agregar los cambios y hacer commit
git add .
git commit -m "Corrige error de contexto de Scaffold usando GlobalKey en RiderDashBoardScreen"
# Subir los cambios a la rama remota
git push --set-upstream origin juan/scaffold-context-error-fix

🔀 Merge a develop

  • El merge debe hacerse siempre a la rama develop.
  • Antes de hacer merge, asegúrate de actualizar tu rama local con los últimos cambios de develop para evitar conflictos.
  • Pasos recomendados:
# 1. Cambia a develop y actualiza
git checkout develop
git pull origin develop

# 2. Cambia a tu rama de trabajo
git checkout juan/scaffold-context-error-fix

# 3. Trae los últimos cambios de develop a tu rama
git merge develop
# o si prefieres rebase:
# git rebase develop

# 4. Resuelve conflictos si los hay, luego:
git add .
git commit -m "Resuelve conflictos con develop"

# 5. Sube tu rama actualizada
git push

# 6. Haz el merge a develop (por PR/MR o desde consola)
# Ejemplo por consola:
git checkout develop
git merge juan/scaffold-context-error-fix
git push
  • Si trabajas en equipo, se recomienda hacer el merge mediante Pull Request (PR) o Merge Request (MR) para revisión de código.
